## 审查清单

**代码质量：**
- 关注点分离是否清晰？
- 错误处理是否恰当？
- 类型安全是否确保（如适用）？
- 是否遵循 DRY 原则？
- 边界情况是否处理？

**架构：**
- 设计决策是否合理？
- 是否考虑可扩展性？
- 是否有性能影响？
- 是否存在安全隐患？

**测试：**
- 测试是否真正测试了逻辑（而非 mock）？
- 是否覆盖边界情况？
- 需要的地方是否有集成测试？
- 所有测试是否通过？

**需求：**
- 是否满足所有计划需求？
- 实现是否符合规范？
- 是否有范围蔓延？
- 破坏性变更是否已记录？

**生产就绪：**
- 是否有迁移策略（如有架构变更）？
- 是否考虑向后兼容？
- 文档是否完整？
- 是否有明显的 bug？

## 关键规则

**应该做：**
- 按实际严重程度分类（并非所有都是严重问题）
- 具体说明（文件:行号，不要含糊）
- 解释为什么问题重要
- 认可优点
- 给出明确结论

**不要做：**
- 未经检查就说"看起来不错"
- 将吹毛求疵的问题标记为严重
- 对未审查的代码给出反馈
- 含糊其辞（"改进错误处理"）
- 避免给出明确结论
